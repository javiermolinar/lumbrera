# Lumbrera Assets

## Context

Lumbrera brain repos have two content directories: `sources/` (immutable reference material) and `wiki/` (synthesized knowledge pages). There is no support for binary attachments like diagrams, screenshots, or PDFs. Wiki pages that need visuals have no managed place to put them.

The current `INDEX.md` is a single generated file listing both sources and wiki pages. As content grows across three types, a single index becomes noisy and wastes LLM context tokens when only one content type is needed.

## Goals

- Add `assets/` as a third content directory for binary attachments
- Assets follow the same lifecycle as sources: immutable on write, deletable via `lumbrera delete` with cascade cleanup
- Wiki pages link to assets via standard Markdown image/link syntax
- Split `INDEX.md` into three focused generated files
- INDEX.md wiki entries include links to their backing sources

## Constraints

- Assets are not Markdown — no frontmatter, no FTS indexing, no analysis
- Asset writes require a `--file` flag (binary content can't be piped as stdin text)
- Asset deletion scrubs references from wiki pages but never cascade-deletes a wiki page (unlike source deletion where zero sources remaining triggers cascade)
- The `assets/` directory is created on `lumbrera init` (empty, with placeholder ASSETS.md)
- Replacement workflow is delete + write + LLM re-adds links (no transactional replace command)
- Existing v1 brains must be explicitly migrated to v2 via `lumbrera migrate`

## Pre-work: Centralize content root definitions

The prefixes `"sources/"`, `"wiki/"` and kind strings `"source"`, `"wiki"` are hardcoded across ~8 packages:

| Package | What's hardcoded |
|---|---|
| `pathpolicy` | prefix → kind mapping, `.md` extension check |
| `markdown/links` | allowed prefixes for link resolution |
| `brainfs` | `kindForPath` prefix → kind |
| `verify/path_policy` | directory list to walk |
| `verify/documents` | `"sources/"` prefix checks on references |
| `writecmd` | kind-based branching for validation, mutation |
| `deletecmd` | kind-based cascade logic |
| `generate` | root names for index generation |

Adding `"assets/"` means touching every one of these. Refactor first:

**New: `internal/brain/roots.go`** — single source of truth for content roots:

```go
type ContentRoot struct {
    Dir       string // "sources", "wiki", "assets"
    Kind      string // "source", "wiki", "asset"
    Markdown  bool   // true for sources/wiki, false for assets
    Required  bool   // true for sources/wiki, true for assets (created on init)
}

var ContentRoots = []ContentRoot{
    {Dir: "sources", Kind: "source", Markdown: true, Required: true},
    {Dir: "wiki", Kind: "wiki", Markdown: true, Required: true},
    {Dir: "assets", Kind: "asset", Markdown: false, Required: true},
}

func IsContentPrefix(path string) (ContentRoot, bool) { ... }
func AllowedLinkPrefix(path string) bool { ... }
func KindForPath(path string) string { ... }
```

Then each package imports `brain.ContentRoots` or the helper functions instead of hardcoding prefixes. Adding a fourth content type later becomes a one-line addition.

This refactor should land as Phase 0 before any asset-specific work.

## Plan

**`internal/pathpolicy/pathpolicy.go`**
- `NormalizeTargetPath`: add `assets/` as third prefix returning `kind = "asset"`
- Drop the `.md` extension requirement when kind is `"asset"`
- No extension whitelist. Only restriction: reject `.md` files under `assets/` (those belong in `sources/` or `wiki/`).

**`internal/brainfs/brainfs.go`**
- `kindForPath`: add `"asset"` case for `assets/` prefix

**Tests**

### Phase 2: Link resolution — allow linking to assets

**`internal/verify/path_policy.go`**

**`internal/verify/documents.go`**

**Tests**

### Phase 3: Write — create assets

**`internal/writecmd/args.go`**
- Add `--file <path>` flag to options struct and parser

**`internal/writecmd/mutation.go`**
- `inferOperation`: `kind == "asset"` + not exists → `opAsset`. Exists → error (immutable).
- `applyMutation`: `opAsset` reads content from `--file` path, calls `writeRawFile`. No frontmatter, no Markdown analysis.

**`internal/writecmd/validation.go`**
- `validateOptionsForOperation` for `kind == "asset"`:
  - Require `--file`
  - Reject `--title`, `--summary`, `--tag`, `--source`, `--append`
  - Reject if file already exists (immutable)

**`internal/writecmd/write.go`**
- Skip stdin read when `op == opAsset` (content comes from `--file`)
- Skip frontmatter/sources-section checks for assets

**Tests**
- Write asset succeeds, file created
- Write asset to existing path rejected
- Write asset without `--file` rejected
- Write asset with `--title`/`--tag`/etc. rejected
- `.md` file under `assets/` rejected
- Verify passes after asset write

### Phase 4: Delete — scrub asset links from wiki pages

**`internal/markdown/`** — new function:
- `StripAssetReferences(body string, assetPath string) string`
  - Remove `![alt](asset-path)` entirely (image embeds)
  - Remove `[text](asset-path)` entirely (linked references to assets — not unwrapped to plain text, since a bare asset filename is useless)
  - Handle both repo-relative and relative link forms

**`internal/deletecmd/cascade.go`**
- `planCascade`: new branch for `kind == "asset"`:
  - Scan all wiki page bodies for image/links pointing to the deleted asset
  - Scrub those references using `StripAssetReferences`
  - Rewrite affected wiki pages (frontmatter regenerated, sources section untouched)
  - Never cascade-delete a wiki page

**`internal/deletecmd/cleanup.go`**
- Add `cleanAssetFromWiki(ref wikiRef, assetPath string) (wikiRef, error)` mirroring existing `cleanSourceFromWiki`

**Delete command output** lists affected pages:

**Tests**

**New: `internal/migratecmd/`** — `lumbrera migrate` command

Upgrades a v1 brain to v2:
1. Validate current brain is v1 (error if already v2, error if not a brain)
2. Create `assets/` directory
3. Regenerate all generated files in new format (INDEX.md wiki-only, SOURCES.md, ASSETS.md)
4. Update VERSION to `lumbrera-brain-v2`
5. Log CHANGELOG entry: `[migrate] [actor]: Upgrade brain v1 → v2`
6. Run verify to confirm integrity
7. Transaction with rollback on failure

**Mutating commands on v1 brains** (write, delete):
- Detect v1, error with: `brain is v1; run "lumbrera migrate" to upgrade to v2`
- Do not auto-migrate

**Read-only commands on v1 brains** (search, health):
- Continue to work on v1 (they read from .brain/search.sqlite, not generated files)
- Print warning: `brain is v1; run "lumbrera migrate" for full feature support`

**`internal/initcmd/`**:
- New brains init as v2 directly

**Skills and AGENTS.md templates:**

## Open decisions

1. **Asset link scrub on delete** — decided: mechanical scrub only (remove `![alt](path)` entirely, unwrap `[text](path)` → `text`). Keeps the always-valid-after-every-operation invariant. The LLM decides whether surrounding prose needs rewriting and does it through `lumbrera write` afterward.
2. **ASSETS.md when empty** — decided: always generate with "No assets yet." placeholder. Create empty `assets/` directory on init too. Consistent with how `sources/` and `wiki/` are always present.
3. **Health/orphan detection** — decided: track asset links in `document_links` like any other internal link. Orphan detection and broken link detection come for free through existing health machinery.
